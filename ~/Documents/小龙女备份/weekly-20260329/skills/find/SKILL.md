# find

通用搜索方法论。用于用户说"帮我找..."但具体来源未知时。

## 核心流程

```
Need → Clarify → Search → Validate → [Found? Deliver : Expand]
```

**关键原则**：
- 不确定时先问1个澄清问题，不要猜测就开始
- 从窄到宽逐步扩展搜索范围
- 交付前验证信息准确性
- 找不到要报告已尝试的路径

## 搜索扩展五步

1. **明显来源** → 直接查找，已知位置
2. **专业来源** → 领域特定数据库，专业社区
3. **替代查询** → 不同关键词，相关概念
4. **间接路径** → "谁会知道这个？什么链接到这个？"
5. **问人** → 需要更多上下文，不同角度

## 搜索前必须明确

| 要素 | 为什么重要 |
|------|-----------|
| 到底是什么？ | 避免找错东西 |
| 成功标准是什么？ | 怎么知道找到了对的 |
| 限制条件？ | 预算、地点、时间、格式 |
| 已经试过什么？ | 不重复失败路径 |

如果用户表述模糊 → 问**一个**澄清问题，然后开始搜。

## 验证清单

交付前检查：
- [ ] 这真的是用户要的吗？
- [ ] 来源可靠吗？
- [ ] 信息是最新的/有效的吗？
- [ ] 用户需要知道什么注意事项？

不确定时直接说："找到了X，但不确定是否完全符合你的需求。"

## 交付格式

```
✅ FOUND: [找到什么]
📍 WHERE: [来源]
🎯 CONFIDENCE: [高/中/低]
⚠️ CAVEATS: [注意事项，如有]
```

多个结果时：汇总让用户选择。

## 未找到时

报告：
- 已尝试的所有路径
- 找到的最接近替代方案
- 建议的不同方向或需要的更多信息

## 网络搜索工具（重要区分）

### 工具选择

| 工具 | 用途 | 使用场景 |
|------|------|---------|
| `openclaw tools search duckduckgo` | 真正的网络搜索 | 搜索新闻/信息/内容 |
| `web_fetch` | 获取网页正文 | 读取已知的URL |
| `online`（内置） | 读文档 | **不是搜索**，是读官方文档 |

**常见错误**：以为 `online` 可以做搜索，实际上它只能读文档。

### 搜索语法（duckduckgo增强）

```bash
# 精确匹配
"精确短语"

# 排除词
搜索内容 -排除词

# 组合搜索
site:github.com 关键词
site:weixin.qq.com 关键词

# 相关搜索
related:example.com
```

### 搜索后验证

找到结果不代表搜对了，问自己：
1. **来源可信吗？** → 官方文档 > 权威媒体 > 博客 > 论坛
2. **时效对吗？** → 是否最新的，还是过时的？
3. **和我的问题匹配吗？** → 有没有偷换概念？

搜完不验证 = 浪费时间。

## 应用：公众号爆文检索

当需要检索同类爆文时：

### Step 1: Clarify
- "你想找哪种类型的爆文？"（人物故事/干货/热点/情感）
- "有具体的人物或主题吗？"
- "是找结构参考还是内容参考？"

### Step 2: Search（从窄到宽）
1. **明显来源**：微信搜索、搜狗微信
2. **专业来源**：新榜、蝉妈妈、第三方榜单
3. **替代查询**："人物稿 10万+"、"XXX 公众号 爆款"
4. **间接路径**：看号主自己是否有其他平台（知乎/小红书）提及

### Step 3: Validate
- 确认阅读量（10万+/在看1万+）
- 确认发布时间（不要找太旧的）
- 确认内容质量（值得学习）

### Step 4: Delivery
提取爆文的核心要素给用户：
- 标题
- 开头3段的钩子
- 核心故事线
- 结尾Callback结构
